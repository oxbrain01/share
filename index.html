Phần đầu (slide tiếp theo): Tổng quan, giới thiệu — DevSecOps là gì, các loại scan, pipeline, điều kiện cần, 3 bước nhanh, lộ trình 5 phase, trạng thái triển khai, cấu trúc repo, luồng chạy.

§1. Tổng quan — DevSecOps & Lợi ích

  • DevSecOps = tích hợp Security vào toàn bộ vòng đời Dev + Ops. Mục tiêu: Shift Left Security — phát hiện lỗ hổng sớm nhất (khi push/MR).
  • Lợi ích: Phát hiện sớm · Tự động hóa · Tuân thủ · Phản hồi nhanh.

Điều kiện cần trước khi bắt đầu

Người mới triển khai cần có sẵn (hoặc được cấp):

  • GitLab (project có quyền CI/CD) hoặc Jenkins.
  • Runner: GitLab → Settings → CI/CD → Runners; Runner cần tag sonarqube (hoặc đổi tag trong ci/*.yml cho khớp).
  • SonarQube: URL truy cập được từ Runner (vd. https://sonar.mycompany.com). Token tạo tại SonarQube → My Account → Security → Generate Tokens.
  • Variables: Đặt tại GitLab → Settings → CI/CD → Variables (Expand) → Add variable. SONAR_TOKEN bắt buộc bật Mask variable.

Nếu thiếu một trong các mục trên, pipeline sẽ fail ở bước tương ứng — xem slide "Lỗi thường gặp (GitLab)" cuối §4.

Quick Start – Triển khai 3 bước

  1. Chuẩn bị: GitLab (hoặc Jenkins), Runner, SonarQube. Variables: SONAR_HOST_URL, SONAR_TOKEN (masked).

§3. Luồng pipeline (stages & jobs)

Thứ tự stage: test → security-scan → build → image-scan. Trong security-scan, các job chạy song song. Fail Fast: security chạy trước build.

Khi nào pipeline chạy: Trong template, job có rules — chỉ chạy khi có Merge Request hoặc push lên branch mặc định (main/master). Push lên branch khác có thể không chạy đủ pipeline.

§4 (tiếp). Lỗi thường gặp – GitLab Pipeline

Khi pipeline fail lần đầu, kiểm tra theo thứ tự:

Triệu chứngCách xử lý
Job sonarqube-scan fail: connection / 401Kiểm tra SONAR_HOST_URL (đúng URL?), SONAR_TOKEN (đã Mask variable? Token còn hiệu lực?). Runner có ra internet tới SonarQube không.
Job không chạy / "No runners available"Runner có tag sonarqube (hoặc đổi tags: trong ci/sast.yml cho khớp tag Runner của bạn).
secret-detection fail: leak trong file testThêm path/stopwords vào .gitleaks.toml (allowlist) — xem §5.3.
Semgrep báo quá nhiều cảnh báo từ node_modulesTạo .semgrepignore với nội dung: node_modules/, dist/.
Build / image-scan fail: image not foundĐảm bảo job build chạy trước và IMAGE_NAME thống nhất (vd. CI_REGISTRY_IMAGE:CI_COMMIT_SHORT_SHA).

Chi tiết Jenkins: xem slide "Jenkins – Xử lý lỗi thường gặp".

Kiểm tra nhanh sau lần chạy đầu

Sau khi push và pipeline chạy xong, nên xác nhận:

  1. Pipeline: Trang CI/CD → Pipelines — tất cả stage (test, security-scan, build, image-scan) đã chạy; job nào failed thì xem log + slide "Lỗi thường gặp – GitLab".
  2. Security Dashboard: Security & Compliance → Dashboard — có báo cáo SAST (Semgrep), Dependency (Trivy fs), Container (Trivy image).
  3. SonarQube: Mở SONAR_HOST_URL → project tên SONAR_PROJECT_KEY — có kết quả scan và (nếu có) coverage.

Nếu thiếu một trong ba: kiểm tra Variables, Runner tag, và rules (MR / default branch).

§5. Vào trong từng file (code & cấu hình)

Phần này mô tả nội dung bên trong từng file đã liệt kê ở §2. Đọc kèm repo mở sẵn.

5.1 .gitlab-ci.yml · 5.2 SonarQube · 5.3 .gitleaks.toml + secret-detection · 5.4 ci/sca.yml · 5.5 ci/image-scan.yml · 5.6–5.8 Enforcement, DAST, Notification · 5.9 Snyk (triển khai chi tiết).

§5.1 File .gitlab-ci.yml (cửa vào)

File ở root: stages, biến, include ci/*.yml. Chỉnh SONAR_PROJECT_KEY (và biến nếu không dùng Variables).

stages: [test, security-scan, build, image-scan]
variables: { SONAR_PROJECT_KEY: be_secOps, FAIL_ON_SECRET_LEAK: "true", FAIL_ON_CRITICAL: "false" }
include:
  - local: ci/sast.yml
  - local: ci/sca.yml
  - local: ci/image-scan.yml

Job test: unit test + artifact coverage/ cho SonarQube. Test fail vẫn tạo lcov tối thiểu (fallback).

§5.2 SonarQube — sonar-project.properties + job

File sonar-project.properties (root): khai báo projectKey, sources, exclusions, đường dẫn coverage. Lấy SONAR_TOKEN: SonarQube → My Account → Security → Generate Tokens → copy vào GitLab Variables (bật Mask variable).

# sonar-project.properties (ví dụ)
sonar.projectKey=be_secOps
sonar.sources=.
sonar.exclusions=**/node_modules/**,**/dist/**,**/coverage/**
sonar.javascript.lcov.reportPaths=coverage/lcov.info

Job sonarqube-scan trong ci/sast.yml: stage security-scan, needs: [test], chạy sonar-scanner với -Dsonar.host.url, -Dsonar.token (từ Variables).

§5.3 Gitleaks — .gitleaks.toml + job secret-detection

File .gitleaks.toml (root): allowlist path/chuỗi không coi là leak. Job secret-detection: gitleaks detect; nếu FAIL_ON_SECRET_LEAK=true và có leak (sau allowlist) → job exit 1.

# .gitleaks.toml (ví dụ tối thiểu)
[extend]
  useDefault = true
[allowlist]
  paths = ['''src/_devsecops-test/.*''', '''docs/.*''']
  stopwords = ["admin123", "example", "changeme"]

Tạo .semgrepignore (root): mỗi dòng một path, ví dụ node_modules/, dist/ — giảm cảnh báo từ thư mục không cần quét.

§5.4 File ci/sca.yml (Trivy SCA, SBOM)

Trong ci/sca.yml: quét filesystem, cache .trivycache/, report GitLab dependency_scanning.

trivy-fs:
  stage: security-scan
  cache: { key: trivy-db, paths: [.trivycache/] }
  script:
    - docker run ... aquasec/trivy:0.69.2 fs . --severity HIGH,CRITICAL
      --ignore-unfixed --format json -o trivy-fs-report.json
    - docker run ... fs . --scanners vuln --format cyclonedx -o sbom-fs.json || true
  artifacts:
    paths: [trivy-fs-report.json, sbom-fs.json]
    reports: { dependency_scanning: trivy-fs-report.json }

--ignore-unfixed = bỏ CVE chưa có bản vá. SBOM CycloneDX phục vụ supply chain & compliance.

§5.5 File ci/image-scan.yml (Trivy Image)

Job trivy-image: stage image-scan, needs: [build] (chạy sau build). Dùng cùng IMAGE_NAME với job build (vd. ${CI_REGISTRY_IMAGE:-$CI_PROJECT_NAME}:${CI_COMMIT_SHORT_SHA}). Mount docker.sock để Trivy đọc image; nếu image ở Registry private thì truyền CI_REGISTRY_USER, CI_REGISTRY_PASSWORD.

trivy-image:
  stage: image-scan
  needs: [build]
  before_script:
    - export IMAGE_NAME="${CI_REGISTRY_IMAGE:-$CI_PROJECT_NAME}:${CI_COMMIT_SHORT_SHA}"
  script:
    - docker run --rm -v /var/run/docker.sock:/var/run/docker.sock ...
      aquasec/trivy:0.69.2 image "${IMAGE_NAME}" --severity HIGH,CRITICAL
      --format json -o trivy-image-report.json
  reports: { container_scanning: trivy-image-report.json }

§5.6–5.8 Enforcement, DAST, Notification

  • Secret: FAIL_ON_SECRET_LEAK=true → Gitleaks exit 1 khi có leak (sau allowlist) → job fail.
  • CRITICAL: FAIL_ON_CRITICAL=true → Semgrep bỏ || true; Trivy thêm --exit-code 1, không || true.
  • allow_failure: Trong template nhiều job đặt allow_failure: true → job fail (cam) nhưng pipeline vẫn passed (xanh). Muốn pipeline đỏ khi CRITICAL/secret thì đặt allow_failure: false cho job tương ứng (hoặc bỏ rule allow_failure khi bật enforcement).
  • Rollback: Tạm set FAIL_ON_CRITICAL=false hoặc FAIL_ON_SECRET_LEAK=false, tạo issue, fix trong 48h, bật lại.
  • 5.7 DAST: Thêm stage deploy-staging, dast; file ci/dast.yml (ZAP), DAST_WEBSITE.
  • 5.8 Phase 5: Job .post + SLACK_WEBHOOK_URL; cache; IaC/License optional.
# Ví dụ trong job (ci/sast.yml)
if [ "${FAIL_ON_CRITICAL}" = "true" ]; then
  semgrep scan --config=auto --gitlab-sast --gitlab-sast-output=gl-sast-report.json .
else
  semgrep scan ... || true
fi

§7. Jenkins (nếu dùng Jenkins)

Yêu cầu & thứ tự stages

Yêu cầu: Agent có Docker (socket mount + Docker CLI trong container). Plugin: Pipeline, SonarQube Scanner; cấu hình SonarQube servers + SonarScanner tool.

Check Docker → SCM → Test → SonarQube → Secret Detection → SAST (Semgrep)
  → IaC Scan → Trivy FS → Build → Image Scan → (Push) → Archive

Security stages chạy trước Build; Image Scan sau Build.

Jenkins – Cấu hình một lần (server)

  • SonarQube Scanner: Manage Jenkins → Global Tool Configuration → SonarQube Scanner (Install automatically hoặc path).
  • SonarQube server: Manage Jenkins → System → SonarQube servers → Add (URL, Server authentication token từ Credentials).
  • Credential SonarQube: Credentials → Secret text (token).
  • Credential Registry (nếu push): Username/Password → ID vd. registry-credentials.

Trong Jenkinsfile: withSonarQubeEnv() lấy URL + token từ cấu hình server.

Jenkins – Tạo job Pipeline

  1. New Item → Pipeline → Pipeline from SCM → Git (URL, Credentials, Branch).
  2. Script Path: Jenkinsfile.
  3. Set biến: WORKSPACE_HOST_PATH (path trên host của workspace hoặc repo), SONAR_PROJECT_KEY.

Repo cần có Jenkinsfile tại root (copy từ be_sec-ops / cicd-devops).

Jenkins – Code: Check Docker & wsPath

Mọi bước mount dùng wsPath = WORKSPACE_HOST_PATH hoặc WORKSPACE. Khi Jenkins trong Docker, host cần path thật có code.

stage('Check Docker') {
  env.DOCKER_CMD = sh(script: 'docker info ... && echo docker', returnStdout: true)?.trim() ?: 'docker'
}
def wsPath = env.WORKSPACE_HOST_PATH?.trim() ?: env.WORKSPACE
// Mọi docker run: -v "${wsPath}:/app" hoặc -v "${wsPath}:/src"

WORKSPACE_HOST_PATH đúng → Test/Gitleaks/Trivy thấy package.json, .git, lockfile.

§7. Jenkins – Code (tóm tắt)

wsPath = WORKSPACE_HOST_PATH hoặc WORKSPACE; mọi docker run dùng -v "${wsPath}:/app" hoặc :/src.

Stages: Test (pnpm test:cov, fallback coverage) → SonarQube (withSonarQubeEnv, qualitygate.wait) → Secret (Gitleaks, FAIL_ON_SECRET_LEAK) → Semgrep (gl-sast-report) → IaC (Trivy config) → Trivy FS (sbom-fs, --exit-code 1 khi FAIL_ON_CRITICAL) → Build → Image Scan (trivy image) → Push (withCredentials nếu REGISTRY_*) → archiveArtifacts.

§5.9 Snyk – Triển khai chi tiết (Open Source + Code)

Snyk gồm hai phần: quét dependency (SCA) và quét mã nguồn (SAST). Dùng CLI, extension IDE (VS Code/Cursor) hoặc job CI (tùy chọn).

Chuẩn bị

  1. Tài khoản snyk.io, API Token (Account/Org settings).
  2. CLI: npm install -g snyk rồi snyk auth (hoặc set SNYK_TOKEN trong CI).

Snyk Open Source (SCA)
